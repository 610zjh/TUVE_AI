## 上线后 90 天的"长期"反馈 / 90-Day Long-Term Feedback

90 天后回访 8 位客户（10 位访谈过的，2 位流失了——和这件事无关，是商业原因），结果：

- 6 位说"明显感觉好用了"
- 1 位说"差不多"
- 1 位说"比之前慢了一点点（但他自己也不确定）" → 我们查了他的账户具体性能数据：实际 P95 = 320ms，比上线前快。猜测是"主观感觉" 偏差。

新增反馈：
- 3 位主动提"希望能自定义首页放哪些数字" → 和 PRD 阶段的 N=1 不同，3 位 = 高共识，可以立 PRD-0048 跟进

## 团队复盘 / Team Retrospective

上线 8 天后开了一次复盘会（参考 [`workflows/operations/incident_response_workflow.md`](../../workflows/operations/incident_response_workflow.md) 复盘格式，虽然这不是事故）。

### 哪里做对了 / What Went Right

1. **澄清问题前置（W1）** —— 工程 lead 列 7 个问题让产品先答，省了至少 1 周返工
2. **性能负载测试提前（W2）** —— 提前发现 P95 不达标，启动了数据预聚合方案，避免了"上线倒逼"
3. **客户原型校准（W3）** —— 真实客户调整了 7 个数字的具体清单，增强了 PRD 验收的精准度
4. **三道闸抓住 PRD 编号** —— 客户面文案 grep 闸是有用的，不是装饰
5. **顺手重构被退回** —— 拆 PR 让 dashboard 和通知重构解耦，rollback 风险降低

### 哪里做错了（系统性）/ What Went Wrong (Systemic)

1. **PRD 第一版的 AC-3 性能目标"P95 ≤ 800ms"是产品凭感觉定的，工程没有 push back** —— 实际上 800ms 在大客户数据规模下是不达标的。**教训**：性能类目标必须由工程基于"最大客户数据规模" 估算后再写进 PRD。
2. **顺手重构发生了**（W2 中段）—— 说明 [`code_quality.md`](../../principles/subs/code_quality.md) §9 的"不要顺手重构" 准则没在工程 lead 的肌肉记忆里。**教训**：在团队 onboarding 文档里把 §9 强调一次。
3. **客户面文案差点漏 PRD 编号** —— 说明工程同事写客户面文案时没默认走红线 #2 流程。**教训**：客户面文案三道闸加进 PR 的"必过项"清单。

### 行动项 / Action Items

- [ ] 工程：性能类 PRD 验收标准必须基于"最大客户数据规模" 估算 → 加进 [`prd_to_implementation.md`](../../workflows/engineering/prd_to_implementation.md) 开工前清单
- [ ] 团队：每月 onboarding 时把"不要顺手重构"作为典型反例讲一次
- [ ] CI / 模板：客户面文案 PR 自动跑 `grep -E '(PRD-|Bug-|内部代号)'` 闸

---

## 教学点 / Teaching Points

- 验收标准要可被打 ✅ / ❌ 不只是有，还要数据撑得住
- 30 天 + 90 天**两次**回访，避免"上线就以为成功了"——长期感受才算
- 复盘**不抓个人，抓系统**——每条 What Went Wrong 都有"系统性教训" + "下次如何防止"
- 行动项必须落到具体动作 + 负责人 + 截止
