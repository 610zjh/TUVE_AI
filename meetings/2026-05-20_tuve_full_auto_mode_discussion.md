## 已确认的需求信号 / Confirmed Requirement Signals

- 信号 1：全托管模式的核心目标是减少用户多轮来回确认，把“问答折磨”压缩成少轮交互。
- 信号 2：用户给出浅需求后，系统应一次性给出尽可能详细的问题清单，优先收齐素材和关键参数。
- 信号 3：用户上传图片、视频、PDF 等素材后，系统需要在内部自动做必要理解，而不是每一步都向用户二次确认。
- 信号 4：在生成视频前，系统必须自检，包括脚本、台词、提示词、素材理解完成度、参数合法性。
- 信号 5：默认体验应偏“能力拉满”，业务经验需持续提醒，例如真人场景优先 `1080p`、默认开配音、默认不加字幕。
- 信号 6：全托管模式仍然保留内部 `1+3 Agent` 流程，只是跳过用户确认门。

## 议题 1: 全托管模式的完整闭环

### 背景 / Context

当前已经定义了全托管模式的基础方向，但还缺完整闭环。若没有闭环，就会出现“能跑但不可验收”的问题。

### 本议题希望确定 / Decision Candidates

- 方案 A：只定义“用户少轮交互”和“自动执行”，不定义验收与回检
- 方案 B：定义完整 loop，覆盖“需求采集 -> 自动执行 -> 质检 -> 交付 -> 修改分流”

### 建议结论 / Recommended

- 采用方案 B。
- 理由：没有验收和回检，就无法判断全托管模式是否真的有效，只能算“自动跑了一次”。

## 议题 2: 质检清单应该包含什么

### 背景 / Context

全托管模式把用户确认门收缩了，系统内部必须用质检清单补回来，否则风险会从“用户觉得麻烦”变成“结果不可控”。

### 建议拆成四道门 / Proposed Gates

- **Gate 1 需求门**：问题清单是否覆盖目标、风格、素材、时长、比例、配音、品牌限制。
- **Gate 2 素材门**：图片/视频/PDF 是否都已完成必要理解；是否存在关键素材缺失；是否有已上传但未消费素材。
- **Gate 3 脚本门**：每个镜头是否有视觉目标、动作、口播或配音文案、参数、素材绑定；是否存在空字段或冲突字段。
- **Gate 4 执行门**：模式选择是否合法、参数是否合法、业务经验是否命中、生成后回检是否通过。

### 建议质检项 / Proposed Checklist

- 需求是否完整到足以写脚本
- 用户历史画像是否已被利用
- 上传素材是否全部盘点
- 需要理解的素材是否真的理解了
- 是否漏了产品图、人物图、品牌资料、PDF 中的核心信息
- 是否有配音文案
- 是否误加字幕
- 是否误用 `720p` 进入真人场景
- 是否存在模式混用
- 是否存在超出数量上限或非法字段

## 议题 3: 如何评测全托管模式

### 背景 / Context

评测不只是“视频好不好看”，还包括模式本身是否降低了沟通成本、是否提高了一次交付率。

### 建议评测维度 / Proposed Evaluation Dimensions

1. **需求理解准确率**
   用户真实意图是否被抓住，问题清单是否问到了关键点。
2. **素材消费完整率**
   用户给的图片、视频、PDF 是否被有效利用，没有遗漏关键素材。
3. **脚本质量**
   结构是否合理，镜头是否清楚，口播和画面是否匹配。
4. **参数与模式正确率**
   是否选对首尾帧模式或参考模式，参数是否合法。
5. **成片质量**
   人脸真实度、画面稳定性、音画同步、信息传达清晰度。
6. **一次交付率**
   用户无需大改即可接受的比例。
7. **沟通压缩率**
   是否将多轮确认压缩到目标轮次。
8. **重试成本控制**
   是否避免无意义自动重跑。

### 建议评分框架 / Proposed Scoring

- 需求理解准确率：15
- 素材消费完整率：10
- 脚本质量：20
- 参数与模式正确率：15
- 成片质量：25
- 一次交付率：10
- 沟通压缩率：5

### 一票否决项 / Hard Failures

- 模式混用
- 非法参数
- 关键素材漏用
- 真人视频误用低清分辨率导致明显假脸
- 配音应该有却没生成

## 议题 4: 如何验收

### 背景 / Context

如果没有验收定义，后续实现完成也无法判断“是模式有问题”还是“执行细节有问题”。

### 建议验收分层 / Proposed Acceptance Layers

- **流程验收**：是否实现少轮交互、内部自动执行和内部自检。
- **质量验收**：评分是否达标，是否无一票否决项。
- **体验验收**：用户是否能在不过度折腾的情况下拿到可接受视频。
- **异常验收**：`needs_repair`、缺素材、弱需求场景是否能稳定回退。

### 建议通过标准 / Proposed Acceptance Criteria

- 总分 >= 80
- 无一票否决项
- 用户交互轮次符合预期
- 小修改可在不重开问题清单的情况下完成
- 大修改能顺利退回先审后发模式

## 议题 5: 首尾帧模式和参考图片模式的区别

### 背景 / Context

当前运行时约束已经明确两种模式不能混用，但业务侧和产品侧仍需要更容易理解的口径。

### 建议口径 / Proposed Framing

- **首尾帧模式**：适用于用户明确要求“第一帧长什么样、最后一帧长什么样”，强调起止画面的强约束。
- **参考图片模式（全能参考模式）**：适用于用户提供参考图、视频、音频，希望模型综合参考而不是锁死首尾帧。

### 判定原则 / Proposed Rule

- 用户明确提到首帧、尾帧、开头、结尾画面锁定，才进入首尾帧模式。
- 只要核心诉求是“参考这个人物/商品/风格/运镜/音频”，默认优先参考模式。
- 同一镜头不得混用。
- 判不清时，优先停留在脚本层澄清，不要直接下发。

## 议题 6: 是否生成关键帧的原则

### 背景 / Context

关键帧不是越多越好，滥用会带来风格漂移，也会增加额外成本和复杂度。

### 建议原则 / Proposed Principles

- 用户核心素材不足，但任务又需要强开场锚点或过渡锚点时，可以生成关键帧。
- 用户素材已经足够且一致时，不额外生成关键帧，优先直接利用真实素材。
- 关键帧是“补缺口”，不是“替代用户给的核心素材”。
- 涉及真人时，关键帧质量不过关不得进入视频生成阶段。

## 业务经验提醒区 / Business Heuristics

> 本区是经验值，不是绝对定律；但在没有反例前，应作为默认提醒反复检查。

- 真人场景优先 `1080p`，避免 `720p` 造成明显假脸。
- 默认开启配音。
- 默认不加字幕，除非用户明确提出或场景明确需要。
- 做真人、商品、品牌视频时，优先检查是否有最核心的商品图或人物图，不要只拿边缘素材开跑。
