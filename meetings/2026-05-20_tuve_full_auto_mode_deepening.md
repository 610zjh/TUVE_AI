## 议题 1: 成本闸门与执行可观测性

### 背景 / Context

原讨论定义了"需求采集 → 自动执行 → 质检 → 交付 → 修改分流"的完整 loop，但在"自动执行"前缺少成本预估节点。全托管模式下用户对花钱无感知，这是最大的信任风险。同时，自动执行不等于黑盒——行业内的 Agent 产品（Devin、Cursor Agent、Manus）都提供执行过程可见但不需操作的"执行看板"。

### 本议题希望确定 / Decision Candidates

- 成本闸门放在哪个节点：脚本定稿后 / 执行前（即 Gate 3 和 Gate 4 之间，记为 Gate 3.5）
- 成本闸门的粒度：预估视频生成次数 × 图片生成次数 × token 消耗区间
- 执行看板的形式：是否需要单独的 UI 层，还是复用现有对话流的进度消息

### 建议结论 / Recommended

- **成本闸门**：在脚本定稿后、首帧生成前插入 Gate 3.5，自动计算本次任务预计消耗（视频生成次数、图片生成次数），超出用户剩余额度或预设上限时暂停，给用户"确认执行（消耗 X credits）"的单项确认。
- **执行可观测性**：每个 shot 生成完成后推送进度消息（"shot 1/8 已完成"），`needs_repair` 触发时推送状态但不中断整体流程（单 shot 降级策略见议题 7）。
- **不要求**本期上线独立执行看板 UI，先走对话流进度消息。

## 议题 2: 质检清单补全——跨镜头一致性、Prompt 可执行性、内容安全

### 背景 / Context

原讨论定义了四道门和 10 条质检项，方向正确但遗漏了三个在视频生成行业内公认不可跳过的大项：跨镜头一致性（当前 AI 视频最大质量瓶颈）、Prompt 反模式匹配（已知必炸的组合拦在生成前）、内容安全与合规（商业产品上线绕不开）。

### 建议在 Gate 3（脚本门）中新增

| 检查项 | 说明 | 严重程度 |
|---|---|---|
| 跨镜头角色一致性 | 同一角色跨镜头的外观描述（发型、服装、肤色、体型）是否一致；是否所有涉及镜头都绑定了同一参考图 | 一票否决 |
| 跨镜头商品一致性 | 同一商品跨镜头的颜色、形状、logo 位置描述是否一致；是否所有涉及镜头都绑定了同一商品参考图 | 一票否决 |
| 风格 prompt 统一性 | 所有镜头的风格描述（光线、色调、画风）是否一致，是否存在矛盾描述 | 阻断 |
| Prompt 反模式匹配 | 是否包含已知必炸组合（如"特写+全身"同时出现、某些不可靠的 motion 描述）；维护一个反模式库，生成前做规则匹配 | 阻断 |

### 建议在 Gate 4（执行门）中新增

| 检查项 | 说明 | 严重程度 |
|---|---|---|
| 内容安全 | 生成内容是否触发安全审核；是否涉及未授权人物肖像/品牌标识 | 一票否决 |
| 水印与版权合规 | 输出是否包含合规水印/版权标识；是否符合平台输出要求 | 阻断 |

### 更新的完整质检清单

在原讨论 10 条基础上，补充上述 6 条，合计 16 条质检项。其中一票否决项从原来的 0 条（原讨论只有评测的一票否决，质检没有）变为质检层也有 3 条一票否决：跨镜头角色不一致、跨镜头商品不一致、内容安全违规。

## 议题 3: 评测维度细化与权重重新分配

### 背景 / Context

原讨论定义了 8 个评测维度和权重，方向正确，但"成片质量 25 分"权重最高却定义最粗（只有 4 条），且缺少"跨镜头一致性"作为独立维度。

### 建议重新分配

| 维度 | 原权重 | 建议权重 | 变化 | 说明 |
|---|---|---|---|---|
| 需求理解准确率 | 15 | 10 | -5 | 重要但不是视频生成独有的能力 |
| 素材消费完整率 | 10 | 10 | 0 | |
| 脚本质量 | 20 | 15 | -5 | 拆出跨镜头一致性后做减法 |
| 参数与模式正确率 | 15 | 10 | -5 | 并入模式安全底线 |
| **跨镜头一致性（新增）** | — | **15** | +15 | 当前最大质量瓶颈，独立维度 |
| 成片质量（细化） | 25 | 25 | 0 | 权重不变，但拆为 5 个子项各 5 分 |
| 一次交付率 | 10 | 10 | 0 | |
| 沟通压缩率 | 5 | 5 | 0 | |

### 成片质量子维度拆分

| 子维度 | 分值 | 评测方法 |
|---|---|---|
| 单镜头画面质量 | 5 | 清晰度、畸变、伪影的人工/自动评分 |
| 人物真实性 | 5 | 面部自然度、肢体协调性（真人场景专属；非真人场景此分挪至"画面质量"） |
| 运动流畅度 | 5 | 是否有跳帧、闪烁、不自然的形变 |
| 音画同步 | 5 | 配音/口播与画面的时序对齐 |
| 信息传达清晰度 | 5 | 观众能否在无字幕情况下理解核心信息 |

### 一票否决项更新

在原讨论 5 条基础上，新增：
- 同一角色跨镜头出现明显身份不一致（长相、服装、体型变化大到观众能察觉）

## 议题 4: Golden Eval Set 与回归拦截机制

### 背景 / Context

原讨论的验收分层思路对，但缺少 ML 工程最基本的两个实践：固定评测集（每次变更后跑同一组用例对比分数）和回归拦截线（相比上一版本不能倒退多少）。

### 建议方案

#### Golden Eval Set

- **规模**：初始 10-15 个典型场景，覆盖：
  - 真人单品推广（3 个）
  - 商品展示（3 个）
  - 知识/口播类（2 个）
  - 弱需求场景（1 个）——用户只给了模糊需求
  - 缺素材场景（1 个）——用户只给了文字，没给图
  - 多素材混合场景（1 个）——图片+视频+PDF
- **维护**：每个 PRD 实现后跑一次全量评测，记录每个维度的分数。
- **扩展**：线上 bad case 在经过修复后，摘录进 Golden Eval Set，防止同类问题回归。

#### 回归拦截线

- 任何一个评测维度得分不得相比上一稳定版本下降超过 **15%**。
- 任何一条一票否决项从"通过"变为"不通过"，阻断上线。
- 总分不得低于 80（沿用原标准）。

#### 评测执行频率

| 节点 | 动作 |
|---|---|
| PRD 实现完成后 | 全量 Golden Eval Set 评测 |
| 每次模式/参数变更 | 全量 Golden Eval Set 评测 |
| 每周 | 抽样评测（从 Golden Eval Set 随机抽 5 个） |
| 线上 bad case 修复后 | 该 case 加入 Golden Eval Set，跑全量确认无回归 |

## 议题 5: 首尾帧/参考模式判定的安全底线

### 背景 / Context

原讨论的判定规则方向对——"用户明确提到首尾帧才进首尾帧模式，否则默认参考模式"、"同一镜头不得混用"——但缺少一条关键安全底线：当系统无法高置信度判定时，不要猜，要问。

### 建议补充

#### 安全底线

> **当系统无法高置信度判定模式时，在问题清单中主动出一道选择题，用用户自己的话术确认模式，不靠关键词匹配硬猜。**

选择题示例：
> "在开始创作前想确认：您更看重开头和结尾的具体画面（比如'第一帧必须是产品正面、最后一帧必须是 logo 特写'），还是更看重整体风格参考（比如'参考这张图的色调和氛围来拍整个视频'）？"

#### 参考模式下素材绑定可见性

> **参考模式下，用户提供的参考图/视频应被显式绑定到每个镜头。在脚本确认时，向用户展示绑定关系**——"镜头 1 参考了您提供的素材 A，镜头 2 参考了素材 B"——让用户能在脚本阶段发现"我的参考图没用上"。

#### 判定规则更新

在原讨论规则基础上，新增一条：
- 判定置信度低于阈值时，不进入任何生成模式，先在脚本层出选择题让用户确认。
- 判定置信度的标准：用户输入中是否包含"首帧""尾帧""开头画面""结尾画面""第一帧""最后一帧"等明确关键词，且没有同时出现"参考风格""参考色调""类似这种感觉"等弱化关键词。

## 议题 6: 关键帧生成的回检与量化门禁

### 背景 / Context

原讨论的"补缺口不替代核心素材"原则完全正确，但缺少两层可执行的约束：生成后的回检（关键帧与相邻真实素材是否风格一致）和真人关键帧的量化质量门禁（而非主观"不过关"）。

### 建议补充

#### 回检机制

关键帧生成后，走一道轻量回检：
- 关键帧与相邻真实素材在色调、光照、风格上的偏差是否在可接受范围内。
- 回检方式：可用现有图片理解能力做风格描述对比，或抽取关键帧与相邻素材做 embedding 相似度检查。
- 回检不通过的：标记为"风格漂移"，不进入视频生成，改用相邻真实素材替代或跳过该关键帧。

#### 真人关键帧量化门禁

| 检查项 | 通过标准 | 不通过动作 |
|---|---|---|
| 人脸可检测 | `see2ai_face_detection` 返回 `has_face: true` | 不进入视频生成，标记为"人脸检测失败" |
| 人脸数量匹配 | 关键帧中检测到的人脸数量与素材中一致（允许 ±1 误差） | 不进入视频生成，标记为"人脸数量不一致" |
| 分辨率达标 | 关键帧分辨率 ≥ 目标视频分辨率的 80% | 降级使用，标记为"分辨率不足"但仍可进入视频生成 |

## 议题 7: 用户信任、静默回退与模式切换体验

### 背景 / Context

全托管模式减少了用户交互，但有三件事如果不设计好，体验会从"省心"变成"不放心"：(1) 用户不知道系统做了什么决策；(2) 执行中遇到问题不知道怎么处理；(3) 想切回先审后发模式时不知道会丢什么。

### 议题 7.1：交付摘要 / Delivery Summary

#### 建议

第一版视频交付后，附一份简短的"我是怎么做的"摘要：

- 用了什么素材（素材名 + 用于哪些镜头）
- 做了什么关键决策（模式选择、分辨率选择、是否生成关键帧及原因）
- 每个 shot 的生成方式（文生视频 / 参考图 / 首尾帧）
- 标记出任何"降级"或"自动修复"的地方

这份摘要不需要用户批准，纯粹是"让用户知道系统没有瞎搞"。

### 议题 7.2：静默回退策略

#### 建议定义明确的回退动作表

| 场景 | 回退动作 |
|---|---|
| 单个 shot 生成失败 | 自动重试 1 次 → 仍失败则用最接近的可用素材替代 → 标记该 shot 为"降级"，在摘要中注明 |
| 超过 30% shot 失败 | 停止自动执行，转入先审后发模式；向用户报告："已完成 X/Y 个镜头，Z 个镜头遇到问题，需要您看一下"——保留已生成的可用 shot |
| 素材理解不到位 | 不阻塞流程，但降低该素材相关 shot 的质量预期，在摘要中标注"镜头 N 的素材理解可能不完整" |
| 需求本身模糊（问题清单无法收敛） | 不出问题清单，直接引导用户："您的需求我还没有完全理解，我们先一步步确认"——转入先审后发模式 |
| `needs_repair` 触发 | 不自动修 payload；单 shot 场景下标记"降级"继续；多 shot 场景下累计到阈值再转入先审后发 |

### 议题 7.3：全托管与先审后发的切换体验

#### 建议

两种模式之间不是单向的。设计原则：

- **单点降级**：全托管模式中任何需要用户决策的节点，降级为该节点的单点确认（先审后发式），而不是整个流程回退。用户确认完这一个点后，继续全托管跑完。
- **批量降级**：当触发"超过 30% shot 失败"或"需求本身模糊"时，整个流程转先审后发。
- **已生成产物的保留**：降级或切换模式时，已生成的脚本、已完成且可用的 shot 保留复用，不丢弃。
- **用户主动切换**：用户可以在任意时刻说"后面每一步都让我确认"，系统从下一节点起切换为先审后发模式。

## 业务经验提醒区补充 / Additional Business Heuristics

> 以下是对原讨论"业务经验提醒区"的补充。

- 跨镜头角色一致性是当前 AI 视频生成最大痛点，脚本阶段多花 2 分钟检查角色描述一致性，能避免生成阶段 20 分钟的返工。
- Prompt 反模式库是活的——每次线上出现新的"必炸组合"都应录入，而不是靠人脑记住。
- 成本闸门不只是"帮用户省钱"，更是"帮系统避免无意义消耗"——弱需求场景下没有成本闸门，系统可能烧掉大量额度产出不可用视频。
- 全托管模式第一次交付的摘要，比视频本身更能决定用户是否愿意继续用。
