## 现有设计 vs 你描述的全自动托管模式

**现有 Mode 2 (AGENTS.md)**: 简单描述为"触发后单次收集需求、自动跳过所有用户门、直接交付视频卡片"，但没有定义**问题清单的生成逻辑**和**缺失答案的默认策略**。

| 轮次 | 用户动作 | Agent 动作 |

---

## 关键设计建议

### 1. 问题清单的设计是核心难点

问题清单不应该是固定模板，而应该是 **agent 根据意图分析动态生成的**。建议分为两层：

- **必答层**（不超过 3-4 个）：直接影响生成质量的核心问题——比如视频主题/产品是什么、想要什么画面风格、时长/比例、是否需要真人出镜
- **选答层**（带默认值）：锦上添花的——比如是否需要配音/字幕、特定品牌色、参考视频风格、背景音乐偏好

关键原则：**每个选答问题旁边标注 agent 的默认选择**，用户只需看不满意的改一下，而不是全部从零填写。

### 2. 缺失答案的"自主判断"需要规则化

你说"用户没有回答全，agent 自行做判断"——这是最容易翻车的地方。建议：

- 从 `USER.md` 中读取用户画像作为默认值来源（行业偏好、视觉风格、品牌资产等）
- 对没有用户画像兜底的缺失项，agent 用**保守中间值**（比如 720p、16:9、10秒、无真人、自动音频）
- 在第 2 轮回复的开头用一句话概括"以下是我替你做的决定：XXX"，让用户心里有底，但**不需要用户再次确认**

### 3. 与现有 Confirmation Gate 的关系

你现有的 SOUL.md Rule 6 要求"生成动作必须通知用户并获得明确确认"。全自动模式需要**显式豁免这条规则**——在 agent 判断用户进入全自动模式后，Rule 6 的确认门被"调用工具本身产生的确认卡片"替代，而不是文字确认。这个豁免逻辑需要写清楚，否则 agent 会在两条矛盾的指令间卡住。

### 4. needs_repair (exit code 2) 在全自动模式下怎么办

这是你当前架构里一个特别重要的机制——视频生成脚本返回 `needs_repair` 时要停下来让用户选 A/B/C。在全自动模式下，建议：

- agent **自行选择最合理的候选方案**（根据 Creative Planner Agent 的判断）
- 最多重试 1 次
- 如果重试后仍然 needs_repair，这时候才打断用户，退回到手动模式

### 5. 触发方式建议

除了现有的关键词触发（"直接生成视频"），建议增加：

- 用户发送的消息**不超过 2 句话 + 没有上传任何素材**时，agent 主动提示"是否开启全自动模式"
- 在 UI 层面可以考虑加一个"一键生成"按钮（这是产品层面的，不影响 agent 侧设计）

### 6. 第 1 轮问题清单的呈现形式

建议用**编号列表**而不是表格，因为用户在手机端回答时，编号列表更容易对照回复"1. xxx 2. xxx"。选答题带默认值标注，比如：

> **1.** 视频的主题/产品是什么？
> **2.** 你想要什么画面风格？（写实 / 动画 / 混合）
> **3.** 时长偏好？ *(默认: 10秒)*
> **4.** 画面比例？ *(默认: 16:9 横屏)*
> **5.** 需要真人出镜吗？ *(默认: 否)* 如果需要，请提供参考照片
> **6.** 需要配音/旁白吗？ *(默认: 背景音乐，无旁白)*

---

## 潜在风险提醒

1. **质量预期管理**：两轮压缩意味着 agent 要做更多自主决策，生成结果可能和用户心目中的差距更大。建议在第 2 轮开始执行时告诉用户"如果效果不满意，可以在此基础上调整"
2. **成本控制**：全自动模式可能导致更多的 needs_repair 重试，尤其是 agent 自行补全的决策不太准时。需要设上限（比如最多 1 次自动重试）
3. **与 Review-First 模式的切换**：用户可能在全自动模式途中觉得不对，需要有优雅的退出路径回到 Review-First 模式

---

总结：你的两轮全自动模式设计方向是对的——核心创新点在于**把"多轮信息收集"压缩成"一次结构化问卷"，把"多次确认"替换为"工具卡片一次确认"**。落地时最关键的是把问题清单的动态生成逻辑和缺失答案的默认策略写细，以及处理好和现有 Rule 6 确认门的豁免关系。你觉得这些方向对不对？有没有哪个点想深入讨论的？
